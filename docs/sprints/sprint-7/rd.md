## Audio Generation Approach

### Shared Audio Format

All sound types share the same `AudioFormat`:
- Sample rate: 44100 Hz
- Bit depth: 16-bit signed PCM
- Channels: 1 (mono)
- Byte order: little-endian

Buffer size: 2048 samples per write (~46.4 ms per iteration). This is small enough that volume and stop changes are perceived as nearly instantaneous.

### White Noise

Each sample is a uniform random value from `kotlin.random.Random.nextFloat()` mapped to `[-1.0, 1.0]` and scaled by the current volume. Flat spectral density across all audible frequencies; effective as a broadband masking signal.

```
sample = (Random.nextFloat() * 2f - 1f) * volume
```

### Brown Noise

Generated by numerical integration of white noise. A running accumulator (`brownAccum`) sums successive white noise samples. Because a pure integrator will drift unboundedly, a soft clamp (`coerceIn(-16.0, 16.0)`) keeps the accumulator within bounds. The output is normalised by dividing by the clamp limit (16.0) to stay within `[-1.0, 1.0]`.

```
brownAccum = (brownAccum + whiteNoiseSample).coerceIn(-16.0, 16.0)
sample = (brownAccum / 16.0) * volume
```

The integrator is held as local state in the audio loop closure — it is reset to zero on each new generator instantiation (i.e., when the user switches sound type or restarts).

### Rain

White noise is multiplied by a slow sinusoidal amplitude envelope oscillating at 1.8 Hz. The envelope range is `[0.25, 1.0]` (never fully silent), which gives the effect of varying rain intensity bursts. The envelope is computed as:

```
envelope = 0.625 + 0.375 * sin(2 * PI * 1.8 * t)
sample = whiteNoiseSample * envelope * volume
```

The time variable `t` is derived from a monotonically incrementing `sampleClock` counter to maintain phase continuity across buffer boundaries.

### Binaural Focus

Two sine waves are summed:
- Wave 1: `sin(2 * PI * 200.0 * t)`
- Wave 2: `sin(2 * PI * 240.0 * t)` (200 + 40 Hz beat offset)

Their sum is halved to remain within `[-1.0, 1.0]`:

```
mixed = (sin(2*PI*200*t) + sin(2*PI*240*t)) * 0.5 * volume
```

The 40 Hz phase difference causes the combined wave's envelope to oscillate at 40 Hz, creating an audible beating effect in the mono output. Phase is maintained via `sampleClock`, ensuring continuity across buffer writes.

## Thread Model

```
[Compose Main Thread]
  AmbientSoundViewModel
    .togglePlayback()          -> creates AmbientSoundGenerator, calls .start()
    .setVolume(v)              -> writes to generator.volume (@Volatile field)
    .selectSound(type)         -> calls stopGenerator(), startGenerator()

[Daemon Thread: "ambient-sound"]
  AmbientSoundGenerator.audioLoop()
    -> opens SourceDataLine
    -> loop: check `running` flag, synthesize BUFFER_SAMPLES, write to line
    -> on stop: drain(), close() the SourceDataLine
    -> on exception: call onError() lambda, exit
```

Key thread safety properties:
- `running` is `@Volatile` — the write from the main thread is immediately visible to the audio thread.
- `volume` is `@Volatile` — same guarantee; no lock needed for a single Float read.
- `generator` reference in the ViewModel is only written on the main thread (Compose single-threaded model for state writes).
- `onError` writes to `isPlaying` and `errorMessage` Compose state from the audio thread. Compose's snapshot system accepts writes from any thread; the state is committed and the UI will recompose on the next frame.

## Volume Control Implementation

Volume is applied as a linear amplitude multiplier at sample-generation time (not via `FloatControl` on the `SourceDataLine`). Reasons:
1. `FloatControl.MASTER_GAIN` availability is not guaranteed across all JVM audio drivers.
2. Linear amplitude scaling in the synthesis loop gives deterministic, driver-independent behavior.
3. There is no audible click on volume change because the transition happens at a buffer boundary; the next batch of samples uses the new volume value.

## Trade-offs and Known Limitations

### Linear amplitude volume vs. perceptual dB scaling
Volume is applied as a linear multiplier. Human perception of loudness is logarithmic, so moving the slider from 0.5 to 1.0 will sound like a larger jump than 0.0 to 0.5. A future improvement could map the slider's [0, 1] range to a dB scale (e.g., -40 dB to 0 dB) for more natural control.

### Brown noise soft-clamp produces mild spectral coloration
The `coerceIn` clamp introduces very slight harmonic distortion at high values of the accumulator, modestly flattening the ideal `1/f²` spectrum. For a focus-tool context this is inaudible and inconsequential.

### Rain envelope is a simple sine oscillator
The envelope oscillates at a fixed 1.8 Hz, giving a rhythmic rather than truly random rainfall pattern. A future improvement could use a Perlin noise or irregular LFO for a more naturalistic rain texture.

### Binaural effect requires headphones to perceive fully
The beating effect from summed mono tones is audible on any output, but traditional binaural entrainment is designed for separate left/right channels at slightly different frequencies. The current mono implementation produces an amplitude-modulated tone rather than a true ILD-based binaural beat. This is documented in the UI display name ("Binaural Focus" with no specific clinical claims).

### onError callback is called from the audio thread
The `onError` lambda in `AmbientSoundGenerator` writes to Compose snapshot state from a non-main thread. Compose's `MutableState` is designed to accept writes from any thread (the snapshot will be applied and recomposition scheduled on the next frame). This is consistent with how Compose handles state writes from coroutines on Dispatchers.IO. This is acceptable and correct in the Compose for Desktop context.

### SourceDataLine buffer size
The line is opened with `BUFFER_SAMPLES * 2 * 2` bytes (two buffers ahead). This provides approximately 93 ms of audio latency on stop. If sub-50 ms stop latency is required, the buffer size can be halved at the cost of increased CPU context-switching.
